# Frontend Philosophy

이 문서는 이 프로젝트에서 굳어진 프론트엔드 설계 철학을 다른 Agent나 후속 프로젝트가 바로 재사용할 수 있게 정리한 메모다.

## 핵심 철학

- 한 화면에는 한 가지 핵심 행동만 둔다.
- 사용자는 지금 할 수 있는 행동을 바로 이해해야 한다.
- 예쁜 화면보다 상태가 명확한 화면이 우선이다.
- 첫 렌더에서 틀린 상태를 잠깐 보여주는 것보다, 짧은 로딩이 낫다.
- 읽기 화면과 편집 화면은 분리한다.
- 서버 비용이 있는 환경에서는 요청 하나의 가치를 높인다.

## 디자인 시스템 원칙

- 색은 적게 쓰고 역할 중심으로 쓴다.
- 메인 블루는 선택 상태와 주요 액션에만 쓴다.
- 연한 블루는 보조 배경, 요약 카드, 선택 가능한 상태에 쓴다.
- 초록은 확정, 완료, 성공 상태에만 쓴다.
- 회색은 비활성, 읽기 전용, 준비중 상태에만 쓴다.
- 카드, 버튼, 칩, 탭은 radius와 그림자 톤을 맞춘다.
- 화면 전체 일관성보다 패널 단위 일관성을 우선한다.

## 레이아웃 원칙

- 모바일 우선으로 설계한다.
- 상단에는 현재 컨텍스트만 둔다.
- 본문에는 핵심 객체 하나를 강조한다.
- 하단에는 다음 행동 하나를 강하게 둔다.
- 정보가 많아지면 요약 후 확장 구조를 쓴다.
- 리스트와 편집기를 동시에 크게 두지 않는다.
- 같은 이름, 상태, 총액을 여러 카드에 반복하지 않는다.

## 상태 표현 원칙

- 설명 문구보다 상태 표현이 중요하다.
- 사용자가 이미 알고 있는 설명은 반복하지 않는다.
- 버튼 라벨은 행동 그대로 쓴다.
- 상태 이름은 짧고 단단하게 유지한다.
- 운영자 화면은 구분, 총액, 우선순위가 먼저 보이게 한다.
- 참가자 기준과 정산 수신자 기준을 혼용하지 않는다.

## 컴포넌트 설계 원칙

- 컨테이너는 데이터와 상태를 가진다.
- 패널과 섹션은 렌더링 책임만 가진다.
- JSX 안에서 복잡한 분기 계산을 직접 하지 않는다.
- 먼저 파생 상태를 만들고, 그 상태를 렌더한다.
- 서버가 줄 수 있는 초기 데이터는 서버에서 준다.
- 클라이언트는 mutation과 국소 갱신에 집중한다.

## UX 원칙

- 사용자가 보는 흐름과 운영자가 보는 흐름을 분리한다.
- 모니터링 화면과 편집 화면을 같은 위치에 과하게 섞지 않는다.
- 홈에서는 판단이 쉬워야 하고, 상세 편집에서는 조작이 쉬워야 한다.
- 합산과 분리 기준은 도메인 단위에 맞춘다.
- 정산은 참가자 기준이 아니라 수신자 기준으로 본다.
- 프로필, 신청, 정산처럼 개인 행동이 들어가는 화면은 심리적 부담이 적어야 한다.

## 성능 원칙

- 첫 화면 이후 같은 데이터를 다시 fetch하지 않는다.
- polling은 제한적으로 쓴다.
- 탭이 열려 있을 때만 갱신하거나, 액션 직후에만 갱신한다.
- 전체 refresh보다 부분 갱신을 우선한다.
- 서버 컴포넌트와 클라이언트 컴포넌트의 역할을 분리한다.
- request-based 환경에서는 작은 중복도 누적 비용이 된다.

## 운영 환경 원칙

- Cloud Run 같은 request-based 환경에서는 SSR과 후속 fetch를 동시에 과하게 쓰지 않는다.
- 공용 데이터와 사용자별 동적 데이터를 분리한다.
- 관리자 페이지는 첫 진입용 데이터는 서버에서 준비하고, 상호작용만 클라이언트 fetch로 남긴다.
- 빌드 시점에 DB가 없는 환경을 고려해, 관리자처럼 런타임 의존 페이지는 동적으로 강제할 수 있어야 한다.

## 안티패턴

- 초기 렌더에서 틀린 상태를 보여주고 나중에 수정하는 것
- fetch, mutation, 파생 상태, JSX를 한 컴포넌트에 몰아넣는 것
- 같은 이름, 금액, 상태를 여러 카드에 반복하는 것
- 페이지 진입마다 bootstrap API를 반복 호출하는 것
- 운영자 모니터링과 편집을 하나의 거대한 화면에 섞는 것

## 다른 Agent용 요약

- 모바일 우선
- 상태 명확성 우선
- 서버 초기화 우선
- 중복 정보 제거
- 읽기와 편집 분리
- request-based 환경에서는 요청 수와 중복 연산을 항상 의식
